# DataLab 지역 방문자 지표와 초기 문화점수 계약

> **담당자:** 황찬우
>
> **공개 API:** `GET /cultures/:id/regions`
>
> **원본 서비스:** 한국관광공사 빅데이터 지역별 방문자 수 `DataLabService`

## 1. 목적과 해석 범위

문화 카테고리를 선택했을 때 현재 큐레이션 목록에 포함된 지역을 초기 장소 밀도, 기초·광역지자체 방문 추이와 큐레이션 점수로 정렬한다.

DataLab의 방문자는 정확한 방문 목적을 알 수 없으므로 관광객과 같은 뜻으로 사용하지 않는다. 광역지자체와 기초지자체 수치는 집계 기준이 달라 서로 합산하지 않는다. CulturePath 점수는 방문자 수 원문이 아니라 같은 지역의 동일 요일 변화 지표만 사용한다.

## 2. 기준일과 비교일

```dotenv
DATALAB_REFERENCE_YMD=20210513
DATALAB_COMPARE_DAYS=7
```

- 실행 중 최신 날짜를 자동 탐색하지 않는다.
- 실제 응답이 확인된 기준일을 환경변수로 고정한다.
- 비교일은 기준일에서 7일 전인 같은 요일이다.
- 기준일이 바뀌면 캐시 키도 바뀌므로 기존 7일 fresh 캐시와 충돌하지 않는다.
- `20210513`은 데이터가 실제로 확인된 기준값이다.
- 2026-07-24 제한 smoke에서 `20260723`과 `20260716`은 기초·광역 모두 정상 응답이지만 0건이었다. 호출 한도 때문에 다른 날짜를 탐색하지 않았으며, 더 최근 유효일이 별도로 검증될 때까지 기준값을 유지한다.

## 3. 제품용 방문자 모델

광역 `metcoRegnVisitrDDList`와 기초 `locgoRegnVisitrDDList`를 다음 내부 필드로 정규화한다.

```json
{
  "level": "local",
  "code": "48220",
  "name": "통영시",
  "dayOfWeekCode": "4",
  "dayOfWeekName": "목요일",
  "visitorTypeCode": "2",
  "visitorTypeName": "외지인(b)",
  "visitorCount": 317425.5,
  "baseYmd": "20210513"
}
```

- 모든 코드와 날짜는 문자열로 보존한다.
- `visitorCount`는 소수를 유지하며 누락·`null`·공백·음수·`NaN`·무한대는 거부한다. 실제 숫자 `0`만 0명으로 인정한다.
- 현지인 `1`, 외지인 `2`, 외국인 `3` 구분을 보존한다.
- 점수 입력에는 외지인과 외국인 수의 합을 사용하고 현지인 수는 사용하지 않는다.
- 실제 존재하는 달력 날짜만 허용한다.
- 전체 결과가 여러 페이지면 최대 5페이지, 요청당 최대 1000개까지 수집한다.
- 원본이 실제 반환한 `numOfRows`로 전체 페이지 수를 계산한다.
- `pageNo`·`numOfRows`·`totalCount`가 누락되거나 페이지 간 일관되지 않으면 거부한다.
- `totalCount`가 음수·소수·문자이거나 실제 수집 건수와 다르면 불완전한 응답으로 거부한다.
- 같은 조회 안에서 `level + code + baseYmd + visitorTypeCode`가 중복되면 중복 페이지나 중복 행으로 보고 거부한다.

## 4. 서비스 지역 코드 매핑

DataLab 기초지자체 5자리 코드는 법정동 시·도 2자리와 시·군·구 3자리 조합으로 해석한다.

```text
signguCode = lDongRegnCd + lDongSignguCd
```

전국 코드를 임의로 연결하지 않고 현재 서비스의 큐레이션 지역만 명시적으로 매핑한다.

- 강원특별자치도와 전북특별자치도 개편 전·후 코드를 같은 지역의 alias로 관리한다.
- 전주와 포항처럼 두 자치구로 구성된 서비스 지역은 두 코드 그룹이 모두 있을 때만 방문 지표를 계산한다.
- 여러 자치구를 합한 값은 중복 방문자를 제거한 도시 순방문자 수가 아니라, 동일 방식으로 비교하는 **변화 지표용 합계**다.
- 서울은 기초지자체 합산 대신 광역지자체 응답만 사용한다.
- 코드 그룹 중 일부가 누락되면 불완전한 수치로 점수를 계산하지 않고 전체 큐레이션 fallback을 사용한다.

## 5. 초기 지역 문화점수

사용자 코스·Fork·완주 데이터가 충분하지 않은 초기 단계에서는 다음 가중치를 사용한다.

```text
초기 지역 문화점수 =
  초기 장소 밀도 40%
  + 방문자 추이 30%
  + 큐레이션 30%
```

### 5.1 초기 장소 밀도

현재 `places_cache`는 전국을 완전히 수집한 원본이 아니라 사용자가 조회한 장소의 부분 캐시다. 따라서 캐시 행 수를 장소 밀도로 사용하지 않는다.

이번 단계에서는 기존 지역 카드의 `spotCount`를 명시적인 초기 큐레이션 입력으로 사용하고, 같은 문화 목록 안에서 `0~100` min-max 점수로 변환한다. 모든 지역의 수가 같으면 중립값 `50`을 사용한다.

전국 TourAPI 분류별 적재가 준비되면 이 공급자만 실제 장소 밀도로 교체한다.

### 5.2 방문자 추이

기준일과 7일 전의 외지인·외국인 합을 비교한다.

```text
변화율 = (기준일 방문지표 - 비교일 방문지표) / 비교일 방문지표 × 100
방문 추이 점수 = clamp(50 + 변화율 × 2.5, 0, 100)
```

- 변화 없음은 `50`
- `+20%` 이상은 `100`
- `-20%` 이하는 `0`
- 비교일이 0이고 기준일이 양수면 `100`
- 두 날짜 모두 0이면 `50`

### 5.3 큐레이션

기존 지역 카드의 문화 적합도 점수를 초기 큐레이션 점수로 명시적으로 사용한다.

최종 결과는 반올림한 `0~100` 정수이며 점수 내림차순, 동점이면 `areaCode` 오름차순으로 정렬한다.

사용자 활동 데이터가 충분해지면 계획서의 장기 가중치인 장소 밀도 40%, 방문 추이 30%, 사용자 활동 20%, 큐레이션 10%로 전환한다.

## 6. 캐시와 장애 처리

전용 `data_lab_query_cache` 테이블에 정규화 응답을 JSON으로 저장한다. 인증키와 전체 요청 URL은 저장하지 않는다.

```dotenv
DATALAB_CACHE_ENABLED=true
DATALAB_CACHE_TTL_SECONDS=604800
DATALAB_CACHE_STALE_MAX_AGE_SECONDS=2592000
DATALAB_CACHE_DB_FAILURE_COOLDOWN_SECONDS=30
```

- fresh TTL: 7일
- stale 허용: 저장 시점부터 30일 미만
- 동일 Node.js 프로세스·동일 키: single-flight
- 캐시 조회부터 갱신까지 single-flight 안에서 처리해 지연된 DB miss도 외부 중복 호출을 만들지 않는다.
- MySQL 장애: 외부 API 직통 fail-open
- 외부 장애: 사용 가능한 stale 응답
- stale도 없거나 필요한 지역 코드가 누락됨: 현재 큐레이션 목록과 점수

## 7. 공개 응답 호환성

기존 Flutter `RegionItem`과 같은 배열을 유지한다.

```json
[
  {
    "areaCode": "tongyeong",
    "name": "통영",
    "description": "박경리·청마 유치환의 흔적",
    "spotCount": 9,
    "score": 84
  }
]
```

body에 원본 방문자 수나 점수 breakdown을 추가하지 않는다.

```http
X-Region-Data-Status: HIT | REFRESHED | STALE | BYPASS | CURATED
```

- `HIT`: 네 요청에 필요한 DataLab 응답이 모두 fresh cache
- `REFRESHED`: 하나 이상을 외부 API로 갱신하고 DB에 저장
- `STALE`: 하나 이상을 외부 장애 때문에 stale로 반환
- `BYPASS`: DB를 우회하고 외부 응답 사용
- `CURATED`: 완전한 방문 지표를 만들 수 없어 기존 점수 사용

## 8. 외부 호출 상한과 검증

문화 목록에 기초 지역만 있으면 cold refresh에서 기준일·비교일 각 1회, 총 2개 DataLab query를 사용한다.

서울처럼 광역 데이터가 함께 필요하면 기초 2회와 광역 2회로 최대 4개 query를 사용한다. 각 query의 페이지 수는 원본 전체 건수에 따라 달라질 수 있으므로 제품 서비스는 5페이지 상한을 추가로 적용한다.

자동 테스트는 API 키를 비우고 네트워크를 차단한다. 승인된 live smoke는 자동 재시도 없이 논리 query 최대 4개만 실행하며 인증키와 전체 URL을 출력하지 않는다.

2026-07-24 live smoke는 `20260723`·`20260716`의 기초·광역 조회를 정확히 4회 실행했다. 네 요청 모두 인증과 JSON 응답 처리는 성공했지만 `totalCount=0`이어서 UTF-8 지역명과 최근 데이터 지연은 재확인하지 못했다. 추가 호출은 수행하지 않았다.

## 9. 이번 단계의 제외 범위

- `region_culture` DB 테이블과 CRUD
- 전국 TourAPI 장소 일괄 수집
- 실제 `places_cache` 기반 문화 장소 밀도
- `/regions/:code/spots` 실데이터 전환
- 사용자 활동 기반 가중치
- Flutter 모델과 화면 변경
- 광역·기초 방문자 수 합산
