# CulturePath 팀 역할 및 협업 기준

> 이 문서는 [서비스 계획서](./문화여행_따라가방_서비스_계획서.md)를 기준으로 팀의 담당 범위와 협업 지점을 정리한다.

## 1. 팀 구성

CulturePath(문화여행·따라가방)는 2명이 함께 개발한다.

| 담당자 | 주 담당 영역 | 핵심 책임 |
| --- | --- | --- |
| 임수민 | Frontend / Backend | Flutter 및 Node.js 기반 서비스의 기본 구조와 CRUD 기능 구현 |
| 황찬우 | API Integration / AI Backend / Frontend Design | 외부 API 연동, RAG 기반 AI 코스 변형 기능 및 서비스 UI/UX 디자인 |

## 2. 임수민 담당 범위

### Frontend 구현

- Flutter 프로젝트의 기본 구조와 공통 화면 흐름 구현
- 홈, 카테고리 상세, 지역 상세, 관광지 상세, 코스 빌더, 코스 상세, 프로필 화면 구현
- Riverpod 상태 관리와 `go_router` 기반 라우팅 구성
- 합의된 내부 API 명세를 바탕으로 CRUD 화면과 사용자 흐름 구현
- 코스 생성·수정, 관광지 담기, Fork, 피드·랭킹 등 일반 서비스 기능 구현
- 황찬우가 전달한 디자인 시안과 디자인 시스템을 Flutter UI에 반영

### Backend 및 데이터베이스 구현

- Node.js/Express REST API의 기본 구조 구성
- MySQL 스키마와 일반 서비스 데이터 관리
- 회원가입·로그인·게스트 이관 등 인증 기능 구현
- 문화, 지역, 장소, 코스 등 기본 CRUD API 구현
- 외부 API를 통해 수집·가공된 관광 데이터를 일반 서비스 기능에 연결
- 코스 Fork, 피드·랭킹, 완주 인증 등 일반 비즈니스 로직 구현

### 주요 담당 API

- `POST /auth/register`, `POST /auth/login`, `POST /auth/migrate-guest`
- `GET /cultures`, `GET /cultures/:id/regions`
- `GET /regions/:code/spots?culture=`
- `GET /places/search`, `GET /places/:id`, `GET /places/:id/related`
- `POST /courses`, `PUT /courses/:id`, `POST /courses/:id/fork`
- `GET /courses/feed`, `GET /courses/ranking`
- `POST /courses/:id/checkin`

## 3. 황찬우 담당 범위

### 외부 API 연동

- 한국관광공사 TourAPI 4.0의 인증, 호출 및 응답 가공 로직 구현
- `searchKeyword2`, `areaBasedList2`, `locationBasedList2`, `detailCommon2`, `detailIntro2`, `detailImage2` 등 관광정보 API 연동
- 관광빅데이터(데이터랩), 연관 관광지, 무장애 여행, 반려동물 동반여행 및 다국어 API의 단계별 연동
- API 서비스키와 환경변수를 클라이언트에 노출하지 않고 백엔드에서 관리
- 외부 API의 호출 제한, 오류, 타임아웃 및 재시도 정책 설계
- TourAPI 응답의 정규화와 `places_cache` 적재·갱신 로직 구현
- 외부 데이터와 내부 문화·지역·장소 식별자를 연결하는 매핑 기준 관리
- OpenRouter를 비롯한 AI 관련 외부 API 연동

### RAG 및 AI Backend

- RAG 파이프라인의 전체 구조 설계 및 구현
- Qdrant 벡터 데이터베이스의 컬렉션·메타데이터·검색 전략 설계
- TourAPI 캐시 및 코스 데이터를 RAG 검색 문맥으로 가공
- OpenRouter LLM 연동과 프롬프트 설계
- 기간, 날씨, 식이, 동행 조건을 반영한 자연어 코스 변형 기능 구현
- 코스 변형 결과의 변경점(diff)과 근거를 반환하는 응답 구조 설계
- AI 세션·메시지 이력 저장 및 일반 백엔드와의 연동

### 주요 담당 API

- `POST /ai/transform`

### Frontend 디자인

- 서비스 전체 UI/UX 방향과 디자인 시스템 수립
- Figma Make를 활용한 화면 구성, 시안 제작 및 반복 개선
- 핵심 사용자 흐름인 `문화 → 지역 → 관광지 → 코스`의 화면 경험 설계
- 홈, 카테고리 상세, 지역 상세, 코스 빌더·상세, AI 채팅 화면 디자인
- 문화 카테고리별 무드 컬러와 Track 셋리스트 타임라인 디자인
- Flutter 구현에 필요한 컴포넌트 상태, 간격, 컬러, 타이포그래피 명세 전달

## 4. 디자인 작업 기준

- 브랜드 무드는 책·문화·골목에서 느껴지는 따뜻하고 문학적인 감성을 지향한다.
- 기본 팔레트는 딥 네이비 `#2B2D42`, 웜 크림 `#F7F3E9`, 테라코타 `#C75B39`, 머스타드 골드 `#D9A441`, 차콜 `#1E1E1E`을 사용한다.
- 제목은 Noto Serif KR, 본문은 Pretendard 또는 Noto Sans KR을 우선 사용한다.
- Figma Make 결과물을 그대로 확정안으로 간주하지 않고, 실제 Flutter 구현 가능성과 반응형 동작을 확인해 보완한다.
- 디자인 전달 시 기본 화면뿐 아니라 로딩, 빈 화면, 오류, 선택, 비활성화 상태를 함께 정의한다.

## 5. 공동 협업이 필요한 지점

| 연동 지점 | 임수민 | 황찬우 | 완료 기준 |
| --- | --- | --- | --- |
| 디자인 핸드오프 | Flutter 구현 가능성 검토 및 반영 | Figma Make 시안·컴포넌트 명세 전달 | 주요 화면과 상태가 Flutter에서 일관되게 동작 |
| AI 변형 API | 코스·장소·사용자 데이터를 제공하고 결과를 저장 | RAG 검색, LLM 호출, 변형 결과 생성 | 요청·응답 스키마에 맞춰 실제 코스가 변형됨 |
| AI 채팅 화면 | Flutter 화면 및 API 연결 | 대화 UX와 diff 미리보기 설계 | 사용자가 결과를 확인한 뒤 적용 또는 취소 가능 |
| 외부 관광 API | 수집된 데이터를 사용하는 CRUD·서비스 로직 구현 | TourAPI 호출, 응답 정규화, 캐시 적재 및 오류 처리 | 외부 데이터가 내부 장소·지역 데이터로 안정적으로 제공됨 |
| 데이터 연동 | MySQL의 일반 서비스 데이터 구조 관리 | 외부 데이터 매핑과 RAG 인덱싱용 가공 기준 정의 | 원본 ID와 벡터 메타데이터를 추적 가능 |
| 통합 QA | CRUD와 일반 사용자 흐름 검증 | 디자인 일관성과 AI 결과 품질 검증 | 핵심 시나리오가 처음부터 끝까지 정상 동작 |

## 6. 개발 일정에 따른 역할

### 1개월 차 — 탐색 경험 구축

- 임수민: DB·인증·문화·지역 CRUD API, Flutter 기본 구조 및 탐색 화면 구현
- 황찬우: TourAPI 연동과 관광 데이터 가공, 디자인 시스템 및 홈·카테고리·지역·관광지 핵심 화면 시안 제작
- 공동 목표: `문화 → 지역 → 관광지` 탐색 흐름 완성

### 2개월 차 — 코스와 RAG 구축

- 임수민: 코스 CRUD·Fork·랭킹, 코스 빌더·Track 타임라인 구현
- 황찬우: 외부 API 연동 보완, 빌더·AI 상호작용 디자인, RAG 파이프라인과 AI 변형 API 구현
- 공동 목표: 코스 빌더·Fork·AI 변형 데모 완성

### 3개월 차 — 통합 및 제출

- 임수민: AI 채팅 연동, 완주 인증, 서비스 QA 및 배포 작업
- 황찬우: AI 변형 품질 개선, UI/UX 보완, 발표용 화면과 시연 흐름 정리
- 공동 목표: 시드 코스, 배포본, 시연 및 공모전 제출 자료 완성

## 7. 우선순위 기준

- **P0:** 문화 카테고리 탐색, 관광지 추천·담기, 코스 빌더, Fork, 피드·랭킹
- **P1:** RAG 기반 AI 코스 변형, 완주 인증·스탬프
- **P2:** 혼잡도 회피 및 기타 확장 기능
- 각 단계에서는 다음 단계 기능보다 현재 우선순위의 핵심 사용자 흐름 완성을 먼저 확인한다.

## 8. 협업 원칙

- 기능 구현 전 API 요청·응답과 데이터 소유 주체를 먼저 합의한다.
- 외부 API의 인증·호출·응답 가공은 황찬우가 담당하고, 임수민은 가공된 내부 데이터의 CRUD와 화면 활용을 담당한다.
- 디자인 변경이 데이터 구조나 API 변경을 요구하면 구현 전에 공유한다.
- RAG 기능은 일반 CRUD 로직과 분리하되, 코스·장소 ID를 공통 식별자로 사용한다.
- 작업 단위별로 담당자를 명확히 하고, 서로의 영역을 수정할 때 변경 내용을 공유한다.
- 역할표에 없는 배포, 발표 자료, 시드 데이터 등의 세부 업무는 일정에 맞춰 함께 조율한다.
